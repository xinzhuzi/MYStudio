## 1. 当前配置中心基线

### 1.1 已有能力

当前 MYStudio 的配置体系围绕三个核心实体构建：

| 实体 | 职责 | 存储位置 |
|------|------|----------|
| **VendorConfig** | 供应商基础信息（名称、API Key、Base URL、启停状态） | `studio-config-store` |
| **ModelDefinition** | 模型定义（模型 ID、名称、归属 Vendor、价格、参数范围） | `studio-config-store` |
| **ModelBinding** | 功能到模型的绑定关系（哪种 GenerationJob.kind 用哪个模型） | `studio-config-store` |

配置持久化通过 `studio-config-store.ts` 完成，基于 Electron 的 `store` 库以 JSON 文件形式落盘。

### 1.2 已有 Store 接口概览

```ts
// studio-config-store.ts 核心结构（简化）
interface StudioConfigStore {
  vendors: VendorConfig[];         // 供应商列表
  modelDefinitions: ModelDefinition[];  // 模型定义列表
  modelBindings: ModelBinding[];  // 功能-模型绑定
}
```

这套结构解决了"能配置"的问题，但缺少运行时的动态能力感知与调度。

### 1.3 明确缺口

经过与 Toonflow 的 vendor 脚本和可编程供应商思路对比，当前配置中心存在以下能力缺口：

| 缺口 | 影响 | 优先级 |
|------|------|--------|
| **V1 不执行模型** | V1 配置只有静态描述，无法在运行时判断供应商是否真正支持某功能 | P0 |
| **缺 Feature Router** | 每次调用 AI 能力都硬编码 provider/model，无法统一调度、轮询、降级 | P0 |
| **缺健康检查** | 用户无法在生成前得知 Key 是否可用、模型是否在线，导致生成失败后才知道问题 | P1 |
| **缺能力标签** | 无法根据 GenerationJob.kind 自动匹配到支持该能力的供应商/模型 | P0 |
| **无多 Key 管理** | 单 Vendor 单 Key，无法做负载均衡或 Key 级别的熔断 | P1 |
| **无版本/迁移机制** | 配置结构变更时无法平滑迁移，可能破坏已有项目 | P1 |

### 1.4 参考来源

- **Toonflow vendor 脚本**：通过供应商侧脚本描述能力并执行适配逻辑
- **Moyin 批处理**：Key Manager + 轮询 + 健康优先策略
- **可编程供应商思路**：高级模式下允许用户自定义供应商适配脚本（受沙箱约束）

## 2. VendorCapability Schema

### 2.1 为什么需要能力声明

当前 `VendorConfig` 只描述"供应商是谁"，不描述"供应商能做什么"。这导致：

1. **生成前无法验证**：用户配置了一个不支持视频生成的模型去做视频任务，请求发出后才失败。
2. **无法自动路由**：Feature Router 不知道哪个供应商能处理当前任务类型。
3. **无法展示 UI 提示**：用户界面无法提前灰化不可用的功能按钮。

VendorCapability 的目标：**让配置中心在生成前就能回答"这个供应商能不能做这件事"**。

### 2.2 TypeScript Interface 定义

```ts
/**
 * 供应商能力声明
 * 描述一个供应商（或其下某个模型）所支持的功能范围。
 * 所有字段均为可选——缺省表示"不支持该能力"。
 */
interface VendorCapability {
  /** 文本生成能力（对话、续写、摘要等） */
  text?: {
    maxTokens: number;       // 单次最大输出 token 数
    stream: boolean;         // 是否支持流式输出
  };

  /** 图像生成能力 */
  image?: {
    maxWidth: number;        // 最大宽度（px）
    maxHeight: number;       // 最大高度（px）
    formats: string[];       // 支持的输出格式，如 ["png", "jpg", "webp"]
  };

  /** 视频生成能力 */
  video?: VideoCapability;

  /** 语音合成能力 */
  tts?: TTSCapability;

  /** 视觉理解能力（图片/视频理解） */
  vision?: {
    maxImageSize: number;    // 最大图片大小（bytes）
    formats: string[];       // 支持的输入图片格式
  };
}

/**
 * 视频生成能力详情
 */
interface VideoCapability {
  imageReference: boolean;   // 是否支持图片作为参考
  videoReference: boolean;   // 是否支持视频作为参考
  audioReference: boolean;   // 是否支持音频作为参考
  durations: number[];       // 支持的时长列表（秒），如 [5, 10]
  resolutions: string[];     // 支持的分辨率，如 ["720p", "1080p"]
  modes: string[];           // 支持的模式，如 ["standard", "professional"]
  maxBatchSize: number;      // 单次请求最大批量
}

/**
 * 语音合成能力详情
 */
interface TTSCapability {
  voices: string[];          // 可用音色列表
  languages: string[];       // 支持的语言，如 ["zh-CN", "en-US"]
  streamOutput: boolean;     // 是否支持流式音频输出
  maxTextLength: number;     // 单次最大输入文本长度（字符数）
}
```

### 2.3 能力声明格式说明

能力声明附着在两个层级上：

| 层级 | 说明 | 示例 |
|------|------|------|
| **Vendor 级** | 供应商的通用能力上限 | OpenAI 支持 text + vision + image(DALL-E) + tts |
| **Model 级** | 具体模型的能力范围（可窄于 Vendor 级） | gpt-4o 支持 text + vision，但 gpt-3.5-turbo 仅 text |

查询逻辑：先查 Model 级，若无则回退到 Vendor 级。

```ts
// 扩展后的 VendorConfig
interface VendorConfig {
  id: string;
  name: string;
  apiKey: string;
  baseUrl: string;
  enabled: boolean;
  capabilities: VendorCapability;  // 新增：供应商级能力声明
}

// 扩展后的 ModelDefinition
interface ModelDefinition {
  id: string;
  vendorId: string;
  name: string;
  capabilities: VendorCapability;  // 新增：模型级能力声明（覆盖 Vendor 级）
}
```

### 2.4 能力标签与 GenerationJob.kind 的映射关系

```ts
/**
 * 功能到能力标签的映射
 * 用于从 GenerationJob.kind 快速定位需要检查的能力维度。
 */
const KIND_TO_CAPABILITY: Record<string, keyof VendorCapability> = {
  'text-generation':  'text',
  'image-generation': 'image',
  'video-generation': 'video',
  'tts':              'tts',
  'vision-analysis':  'vision',
};
```

**为什么需要映射**：GenerationJob.kind 是业务侧的概念（"我要做什么"），而 VendorCapability 是供应侧的概念（"我能做什么"）。映射表将两者桥接，让 Feature Router 能用统一逻辑完成匹配。

### 2.5 能力不足前置阻断规则

核心原则：**在用户发起生成请求之前就拦截不支持的操作，而非等到请求失败后报错。**

```ts
/**
 * 检查某个供应商/模型是否具备执行指定 kind 任务的能力
 * @returns pass / warning / block
 */
function checkCapability(
  kind: string,
  vendorConfig: VendorConfig,
  modelDef?: ModelDefinition
): { status: 'pass' | 'warning' | 'block'; reason?: string } {
  const capKey = KIND_TO_CAPABILITY[kind];
  if (!capKey) {
    return { status: 'block', reason: `未知的任务类型: ${kind}` };
  }

  // 优先检查 Model 级能力
  const cap = modelDef?.capabilities?.[capKey] ?? vendorConfig.capabilities?.[capKey];

  if (!cap) {
    return {
      status: 'block',
      reason: `模型 ${modelDef?.name ?? vendorConfig.name} 不支持 ${kind}`,
    };
  }

  // 可扩展：检查具体参数范围（如视频时长、图片分辨率等）
  return { status: 'pass' };
}
```

阻断时机：
1. **UI 层**：灰化不可用按钮 + 悬停提示
2. **任务调度层**：Feature Router 调度前二次校验
3. **不阻塞用户配置**：用户可以配置任何模型，只是在实际生成时拦截

## 3. 健康检查协议

### 3.1 为什么需要健康检查

当前用户体验：**用户点"生成"，等了30秒，收到一个模糊的错误提示。** 健康检查的目标是在生成前就把潜在问题暴露出来，将"生成后失败"转变为"生成前预警"。

### 3.2 检查项分级

| 时机 | 检查项 | 级别 | 说明 |
|------|--------|------|------|
| **启动时** | API Key 可用性 | warning | 发送简单 ping 请求或读取历史成功率 |
| **启动时** | FFmpeg 路径 | block | 视频相关功能依赖 FFmpeg |
| **启动时** | 旧数据兼容性 | warning | 检测配置文件版本，缺少字段补默认值 |
| **任务前** | 模型绑定存在性 | block | 当前 kind 对应的模型是否已配置 |
| **任务前** | 能力匹配 | block | 模型是否支持当前 kind |
| **任务前** | Key 额度/速率 | warning | Key 是否接近限额 |

### 3.3 检查结果分级

```ts
type HealthStatus = 'pass' | 'warning' | 'block';

interface HealthCheckResult {
  id: string;               // 检查项唯一标识
  name: string;             // 检查项名称（中文）
  status: HealthStatus;
  message: string;          // 用户可见的描述
  suggestion?: string;      // 修复建议
}
```

分级含义：
- **pass**：一切正常，无需处理。
- **warning**：可用但不理想，提示用户但不阻断操作。例如 Key 历史成功率偏低但未完全失效。
- **block**：不可用，必须修复才能继续。例如 FFmpeg 未安装、模型未绑定。

### 3.4 启动时检查流程

```ts
/**
 * 启动时健康检查
 * 在应用启动或项目打开时执行。
 */
async function runStartupChecks(config: StudioConfigStore): Promise<HealthCheckResult[]> {
  const results: HealthCheckResult[] = [];

  // 1. API Key 可用性
  for (const vendor of config.vendors) {
    if (!vendor.enabled) continue;
    const keyResult = await checkApiKey(vendor);
    results.push(keyResult);
  }

  // 2. FFmpeg 路径
  const ffmpegResult = checkFFmpeg();
  results.push(ffmpegResult);

  // 3. 旧数据兼容性
  const compatResult = checkDataCompatibility(config);
  results.push(compatResult);

  return results;
}

/**
 * API Key 检查策略
 * 优先使用轻量 ping（如 models.list），失败则读取历史成功率。
 */
async function checkApiKey(vendor: VendorConfig): Promise<HealthCheckResult> {
  try {
    // 发送最小化请求验证 Key 可用性
    await pingVendorApi(vendor);
    return {
      id: `apikey-${vendor.id}`,
      name: `${vendor.name} API Key`,
      status: 'pass',
      message: 'API Key 可用',
    };
  } catch (err) {
    // 回退：检查历史成功率
    const successRate = getHistoricalSuccessRate(vendor.id);
    if (successRate > 0.8) {
      return {
        id: `apikey-${vendor.id}`,
        name: `${vendor.name} API Key`,
        status: 'warning',
        message: 'API Key 暂时无法验证，但历史成功率较高',
        suggestion: '如果生成失败，请检查 API Key 是否过期',
      };
    }
    return {
      id: `apikey-${vendor.id}`,
      name: `${vendor.name} API Key`,
      status: 'block',
      message: `API Key 验证失败: ${err.message}`,
      suggestion: '请在配置中更新 API Key',
    };
  }
}
```

### 3.5 任务前检查流程

```ts
/**
 * 任务前检查
 * 在 Feature Router 调度前执行，确保当前任务可以执行。
 */
function runPreTaskChecks(
  kind: string,
  config: StudioConfigStore
): HealthCheckResult[] {
  const results: HealthCheckResult[] = [];

  // 1. 模型绑定存在性
  const binding = config.modelBindings.find(b => b.kind === kind);
  if (!binding) {
    results.push({
      id: `binding-${kind}`,
      name: `${kind} 模型绑定`,
      status: 'block',
      message: `未配置 ${kind} 的模型绑定`,
      suggestion: '请在设置中为该功能绑定一个模型',
    });
    return results; // 绑定不存在，后续检查无意义
  }

  // 2. 能力匹配
  const model = config.modelDefinitions.find(m => m.id === binding.modelId);
  const vendor = config.vendors.find(v => v.id === model?.vendorId);
  const capResult = checkCapability(kind, vendor!, model);
  results.push({
    id: `capability-${kind}`,
    name: `${kind} 能力检查`,
    status: capResult.status,
    message: capResult.reason ?? `${model?.name} 支持 ${kind}`,
  });

  return results;
}
```

### 3.6 用户可见提示格式

检查结果通过 Toast/Dialog 展示给用户：

| 级别 | UI 表现 | 用户操作 |
|------|---------|----------|
| **pass** | 不展示（不打扰用户） | 无 |
| **warning** | 黄色 Toast，5秒后自动消失 | 可忽略 |
| **block** | 红色 Dialog，需用户确认 | 必须处理或跳过 |

### 3.7 与项目打开流程的集成

```
用户打开项目
  └─> 触发 runStartupChecks()
       ├─> 全部 pass → 正常进入项目
       ├─> 有 warning → 显示警告摘要 Toast，不阻断
       └─> 有 block → 弹出配置诊断面板，引导修复
```

## 4. Feature Router 设计

### 4.1 为什么需要 Feature Router

当前问题：**每个 AI 功能调用都硬编码了 provider 和 model**。这意味着：
- 切换供应商需要改多处代码
- 无法实现多 Key 轮询和负载均衡
- 无法做自动降级（A 供应商挂了自动切 B）
- 无法统一管理错误和重试

Feature Router 的目标：**提供一个统一入口，让业务层只需说"我要什么功能"，由 Router 决定"用哪个供应商的哪个模型"**。

### 4.2 统一入口：callFeatureAPI

```ts
/**
 * Feature Router 统一入口
 * 所有 AI 功能调用都通过此函数发起。
 *
 * @param kind - 功能类型，如 'text-generation', 'image-generation'
 * @param payload - 请求参数（根据 kind 不同，结构不同）
 * @param options - 可选的调用选项（覆盖默认路由、指定模型等）
 * @returns 统一的响应结构
 */
async function callFeatureAPI<T = unknown>(
  kind: string,
  payload: unknown,
  options?: FeatureCallOptions
): Promise<FeatureResponse<T>> {
  // 1. 前置检查
  const checks = runPreTaskChecks(kind, getConfigStore());
  const blockers = checks.filter(c => c.status === 'block');
  if (blockers.length > 0) {
    throw new FeatureRouterError(
      'BLOCKED',
      `任务被阻断: ${blockers.map(b => b.message).join('; ')}`
    );
  }

  // 2. 路由决策
  const route = resolveRoute(kind, options);

  // 3. 执行请求（含重试和降级）
  return executeWithFallback(route, kind, payload);
}
```

### 4.3 路由逻辑

```ts
interface Route {
  vendorId: string;
  modelId: string;
  apiKey: string;
  baseUrl: string;
}

interface FeatureCallOptions {
  preferVendor?: string;    // 优先使用的供应商
  preferModel?: string;     // 优先使用的模型
  maxRetries?: number;      // 最大重试次数（默认 2）
  timeout?: number;         // 超时时间（ms，默认 60000）
}

/**
 * 路由决策：根据 kind 和 options 确定使用的供应商和模型
 */
function resolveRoute(kind: string, options?: FeatureCallOptions): Route {
  const config = getConfigStore();

  // 1. 如果用户指定了偏好，优先使用
  if (options?.preferModel) {
    const model = config.modelDefinitions.find(m => m.id === options.preferModel);
    if (model) {
      const vendor = config.vendors.find(v => v.id === model.vendorId);
      if (vendor?.enabled) {
        return { vendorId: vendor.id, modelId: model.id,
                 apiKey: vendor.apiKey, baseUrl: vendor.baseUrl };
      }
    }
  }

  // 2. 从 ModelBinding 查找默认绑定
  const binding = config.modelBindings.find(b => b.kind === kind);
  if (!binding) {
    throw new FeatureRouterError('NO_BINDING', `未配置 ${kind} 的模型绑定`);
  }

  const model = config.modelDefinitions.find(m => m.id === binding.modelId);
  const vendor = config.vendors.find(v => v.id === model?.vendorId);

  if (!vendor?.enabled) {
    // 3. 默认绑定不可用，尝试 fallback
    return resolveFallback(kind, config);
  }

  return { vendorId: vendor.id, modelId: model.id,
           apiKey: vendor.apiKey, baseUrl: vendor.baseUrl };
}
```

### 4.4 多 Key 轮询策略

借鉴 Moyin 批处理的 Key Manager 设计，支持同一供应商下多个 API Key 的管理。

```ts
/**
 * Key Manager
 * 管理同一供应商下的多个 API Key，支持轮询和健康追踪。
 */
class KeyManager {
  private keys: KeyEntry[];
  private currentIndex: number = 0;

  constructor(keys: string[]) {
    this.keys = keys.map(k => ({
      key: k,
      healthy: true,
      lastUsed: 0,
      failureCount: 0,
      successCount: 0,
    }));
  }

  /**
   * 获取下一个可用的 Key
   * 策略：优先健康 Key，同等健康度下轮询
   */
  getNext(): KeyEntry | null {
    const healthyKeys = this.keys.filter(k => k.healthy);
    if (healthyKeys.length === 0) {
      // 所有 Key 都不健康，尝试重置
      this.keys.forEach(k => { k.healthy = true; k.failureCount = 0; });
      if (this.keys.length === 0) return null;
      return this.keys[0];
    }

    // 轮询选择
    const selected = healthyKeys[this.currentIndex % healthyKeys.length];
    this.currentIndex++;
    return selected;
  }

  /**
   * 报告 Key 使用结果
   */
  reportResult(key: string, success: boolean): void {
    const entry = this.keys.find(k => k.key === key);
    if (!entry) return;

    if (success) {
      entry.successCount++;
      entry.failureCount = 0;  // 成功后重置失败计数
      entry.healthy = true;
    } else {
      entry.failureCount++;
      // 连续失败 3 次标记为不健康
      if (entry.failureCount >= 3) {
        entry.healthy = false;
      }
    }
    entry.lastUsed = Date.now();
  }
}

interface KeyEntry {
  key: string;
  healthy: boolean;
  lastUsed: number;
  failureCount: number;
  successCount: number;
}
```

### 4.5 Fallback 降级策略

```ts
/**
 * 降级策略
 * 当主供应商不可用时，依次尝试备选。
 */
function resolveFallback(kind: string, config: StudioConfigStore): Route {
  const capKey = KIND_TO_CAPABILITY[kind];
  if (!capKey) {
    throw new FeatureRouterError('NO_FALLBACK', `未知的任务类型: ${kind}`);
  }

  // 查找所有支持该能力的、已启用的供应商
  const candidates = config.vendors
    .filter(v => v.enabled && v.capabilities?.[capKey])
    .sort((a, b) => {
      // 按历史成功率降序排列
      const rateA = getHistoricalSuccessRate(a.id);
      const rateB = getHistoricalSuccessRate(b.id);
      return rateB - rateA;
    });

  if (candidates.length === 0) {
    throw new FeatureRouterError(
      'NO_PROVIDER',
      `没有可用的供应商支持 ${kind}`
    );
  }

  const vendor = candidates[0];
  // 找到该供应商下支持该能力的模型
  const model = config.modelDefinitions.find(
    m => m.vendorId === vendor.id && m.capabilities?.[capKey]
  );

  return {
    vendorId: vendor.id,
    modelId: model?.id ?? 'default',
    apiKey: vendor.apiKey,
    baseUrl: vendor.baseUrl,
  };
}

/**
 * 带降级的执行
 */
async function executeWithFallback(
  route: Route,
  kind: string,
  payload: unknown,
  maxAttempts: number = 3
): Promise<FeatureResponse> {
  let lastError: Error | null = null;

  for (let attempt = 0; attempt < maxAttempts; attempt++) {
    try {
      const result = await callVendorApi(route, kind, payload);
      return result;
    } catch (err) {
      lastError = err;
      // 标记当前 Key 不健康
      keyManager.reportResult(route.apiKey, false);
      // 尝试降级
      const config = getConfigStore();
      route = resolveFallback(kind, config);
    }
  }

  throw new FeatureRouterError(
    'ALL_KEYS_FAILED',
    `所有供应商均失败。最后错误: ${lastError?.message}`
  );
}
```

### 4.6 错误处理策略

| 场景 | 策略 | 用户感知 |
|------|------|----------|
| 单次请求失败 | 自动重试（最多2次） | 无感知 |
| 单 Key 连续失败 | 标记不健康，切换 Key | 无感知 |
| 所有 Key 失败 | 降级到备选供应商 | 可能有延迟 |
| 所有供应商失败 | 返回明确错误信息 | 需手动处理 |

## 5. 旧配置迁移方案

### 5.1 为什么需要迁移方案

配置结构从 V1 升级到 V2，新增了 `capabilities`、`version`、多 Key 等字段。如果不做迁移：
- 用户升级后旧配置无法识别，所有设置丢失
- 新代码读取旧结构时抛异常或产生 undefined 行为
- 用户被迫手动重新配置所有供应商

迁移目标：**用户升级后打开项目，配置自动升级，无感知。**

### 5.2 V1 到 V2 的配置文件结构变化

```ts
// ========== V1 结构（旧） ==========

  name: string;

  baseUrl: string;
  enabled: boolean;
  // 无 capabilities
  // 无 version
}

  vendorId: string;
  name: string;

interface StudioConfigV1 {

// ========== V2 结构（新） ==========

  id: string;
  name: string;
  apiKeys: string[];          // 改为数组，支持多 Key
```

### 5.3 自动迁移逻辑

```ts
/**
 * 配置迁移器
 * 在打开项目时检测 version 字段，自动迁移。
 */
function migrateConfig(raw: unknown): StudioConfigV2 {
  // 检测版本
  const version = (raw as any)?.version ?? 1;

  if (version === 2) {
    return raw as StudioConfigV2;  // 已是最新，无需迁移
  }

  if (version === 1 || !version) {
    return migrateV1toV2(raw as StudioConfigV1);
  }

  // 未来版本
  throw new Error(`不支持的配置版本: ${version}`);
}

/**
 * V1 → V2 迁移
 */
function migrateV1toV2(v1: StudioConfigV1): StudioConfigV2 {
  return {
    version: 2,
    vendors: v1.vendors.map(v => ({
      ...v,
      apiKeys: [v.apiKey],         // 单 Key 转数组
      capabilities: inferDefaultCapabilities(v),  // 推断默认能力
      version: 2,
    })),
    modelDefinitions: v1.modelDefinitions.map(m => ({
      ...m,
      // capabilities 不填，回退到 Vendor 级
    })),
    modelBindings: v1.modelBindings.map(b => ({
      ...b,
      fallbackModelId: undefined,   // 新字段，暂无降级
    })),
  };
}

/**
 * 根据 Vendor 信息推断默认能力
 * 对于已知供应商（OpenAI、Google 等）使用预设模板，
 * 对于未知供应商默认只有 text 能力。
 */
function inferDefaultCapabilities(vendor: VendorConfigV1): VendorCapability {
  const PRESET_CAPS: Record<string, VendorCapability> = {
    'openai': {
      text:  { maxTokens: 16384, stream: true },
      image: { maxWidth: 2048, maxHeight: 2048, formats: ['png', 'jpg', 'webp'] },
      vision: { maxImageSize: 20 * 1024 * 1024, formats: ['png', 'jpg', 'gif', 'webp'] },
      tts:   { voices: ['alloy', 'echo', 'fable'], languages: ['zh-CN', 'en-US'],
               streamOutput: true, maxTextLength: 4096 },
    },
    // 可继续添加其他供应商预设...
  };

  // 按 baseUrl 或 name 匹配
  const key = Object.keys(PRESET_CAPS).find(
    k => vendor.name.toLowerCase().includes(k) || vendor.baseUrl.includes(k)
  );

  return key ? PRESET_CAPS[key] : {
    text: { maxTokens: 4096, stream: true },  // 最保守默认值
  };
}
```

### 5.4 回滚策略

```ts
/**
 * 迁移前备份
 * 在配置文件同目录下创建 .backup 文件。
 */
function backupConfig(configPath: string): string {
  const backupPath = `${configPath}.v1.backup.${Date.now()}`;

  return backupPath;

/**
 * 完整的迁移流程
 */
async function migrateWithBackup(
  configPath: string
): Promise<StudioConfigV2> {
  // 1. 读取原始配置
  const raw = JSON.parse(fs.readFileSync(configPath, 'utf-8'));

  // 2. 检测是否需要迁移
  if (raw.version === 2) {
    return raw;  // 无需迁移
  }

  // 3. 备份
  const backupPath = backupConfig(configPath);
  console.log(`配置已备份至: ${backupPath}`);

  // 4. 执行迁移
  try {
    const migrated = migrateConfig(raw);
    // 5. 写入新配置
    fs.writeFileSync(configPath, JSON.stringify(migrated, null, 2), 'utf-8');
    return migrated;
  } catch (err) {
    // 6. 迁移失败，恢复备份
    fs.copyFileSync(backupPath, configPath);
    throw new Error(`配置迁移失败，已恢复备份: ${err.message}`);
  }
}
```

### 5.5 向后兼容保证

| 保证 | 说明 |
|------|------|
| **数据不丢失** | V1 的所有字段在 V2 中都有对应，迁移是纯增量 |
| **可回退** | 备份文件保留，用户可手动恢复到 V1 |
| **渐进式** | 新字段都有默认值，旧代码读新结构时不会崩溃 |
| **幂等性** | 多次执行迁移结果一致，已迁移的不会重复处理 |

## 6. 动态 Vendor 安全边界

### 6.1 设计原则

动态 Vendor（可编程供应商）允许用户自定义供应商适配脚本，类似 Toonflow 的 vendor script 思路。但**安全是第一优先级**：

> **默认状态：仅能力描述，不执行任何脚本。**  
> 高级模式需要用户明确启用，并理解安全风险。

### 6.2 高级模式启用条件

### 6.3 脚本沙箱策略

#### 6.3.1 执行环境隔离

```ts
import { IsolatedVM } from 'isolated-vm';

/**
 * 创建沙箱执行环境
 * 使用 isolated-vm 将供应商脚本与主进程隔离。
 */
async function createSandbox(scriptContent: string): Promise<IsolatedVM> {
  // 1. 创建隔离的 VM 实例，限制内存和执行时间
  const isolate = new ivm.Isolate({
    memoryLimit: 32,           // 32MB 内存上限
    snapshot: undefined,       // 不使用快照
  });

  // 2. 创建上下文
  const context = isolate.createContextSync();
  const jail = context.global;

  // 3. 注入受限的全局对象
  jail.setSync('global', jail.derefInto());
  jail.setSync('console', createSandboxedConsole());  // 受限 console

  // 4. 注入允许的网络请求函数（受白名单约束）
  jail.setSync('fetch', createSandboxedFetch());

  // 5. 注入允许的文件系统函数（受限）
  jail.setSync('readFile', createSandboxedFS('read'));
  jail.setSync('writeFile', createSandboxedFS('write'));

  // 6. 编译并执行脚本
  const script = isolate.compileScriptSync(scriptContent);
  script.runSync(context, { timeout: 5000 });  // 5秒初始化超时

  return isolate;
}
```

**为什么选 isolated-vm 而非 VM2**：VM2 已被官方废弃（存在沙箱逃逸漏洞），isolated-vm 基于 V8 Isolate 提供更强的隔离保证。

> **编译注意**：`isolated-vm` 是原生 C++ 模块，在 Electron 中需要使用 `electron-rebuild` 或 `@electron/rebuild` 重新编译。打包时需在 electron-builder 配置中添加：
> ```yaml
> # electron-builder.yml
> npmRebuild: true
> buildDependenciesFromSource: true
> ```
> 如果跨平台编译有问题，备选方案是使用 Electron 的 `utilityProcess`（Electron 21+）将动态脚本放到独立子进程中执行。

#### 6.3.2 权限白名单

```ts
/**
 * 网络请求权限白名单
 * 脚本只能请求在 VendorConfig 中预先声明的域名。
 */
function createSandboxedFetch(vendor: VendorConfigV2) {
  const allowedDomains = vendor.allowedDomains ?? [];  // 新增字段

  return async function sandboxedFetch(url: string, options?: RequestInit) {
    const parsedUrl = new URL(url);
    const isAllowed = allowedDomains.some(
      domain => parsedUrl.hostname === domain ||
                parsedUrl.hostname.endsWith(`.${domain}`)
    );

    if (!isAllowed) {
      throw new Error(
        `域名 ${parsedUrl.hostname} 不在白名单中。` +
        `允许的域名: ${allowedDomains.join(', ')}`
      );
    }

    // 转发到主进程执行实际请求
    return ipcRenderer.invoke('sandbox-fetch', url, options);
  };
}

/**
 * 文件系统访问限制
 * 只允许读写项目目录下的文件。
 */
function createSandboxedFS(mode: 'read' | 'write') {
  const projectDir = getProjectDirectory();

  return function sandboxedFS(path: string, ...args: unknown[]) {
    const resolvedPath = pathResolve(projectDir, path);

    // 路径遍历检查
    if (!resolvedPath.startsWith(projectDir)) {
      throw new Error(`路径越界: ${path} 不在项目目录内`);
    }

    if (mode === 'read') {
      return fs.readFileSync(resolvedPath, 'utf-8');
    } else {
      fs.writeFileSync(resolvedPath, args[0] as string, 'utf-8');
    }
  };
}
```

### 7.3 健康检查 action

**怎么做**：在 store 中新增 `checkConfigHealth` 方法，供 UI 层调用。

**为什么**：将健康检查逻辑内聚到 store 层，UI 只需调用一个方法就能获取完整的配置诊断结果。

```ts
// studio-config-store.ts 新增方法

/**
 * 配置健康检查
 * 在项目打开时或用户手动触发时执行。
 */
async function checkConfigHealth(): Promise<HealthCheckReport> {
  const results: HealthCheckResult[] = [];

  // 启动时检查
  const startupResults = await runStartupChecks(this);
  results.push(...startupResults);

  // 汇总
  const blocks = results.filter(r => r.status === 'block');
  const warnings = results.filter(r => r.status === 'warning');

  return {
    overallStatus: blocks.length > 0 ? 'block' :
                   warnings.length > 0 ? 'warning' : 'pass',
    results,
    summary: {
      total: results.length,
      passed: results.filter(r => r.status === 'pass').length,
      warnings: warnings.length,
      blocks: blocks.length,
    },
  };
}

interface HealthCheckReport {
  overallStatus: HealthStatus;
  results: HealthCheckResult[];
  summary: {
    total: number;
    passed: number;
    warnings: number;
    blocks: number;
  };
}
```

### 7.4 Feature Router 作为独立模块

**怎么做**：新建 `src/lib/ai/feature-router.ts`，不修改现有 store 的核心读写逻辑。

**为什么**：Feature Router 是调度层，不是数据层。将它独立出来实现关注点分离，store 只负责配置的 CRUD，Router 负责使用配置做路由决策。

```
src/lib/ai/
├── feature-router.ts          # 统一入口 callFeatureAPI
├── key-manager.ts             # 多 Key 管理
├── health-check.ts            # 健康检查协议
├── capability-check.ts        # 能力匹配逻辑
└── vendor-sandbox.ts          # 动态 Vendor 沙箱（高级模式）
```

与 store 的交互方式：
- Router 通过 `getConfigStore()` 读取配置（只读）
- Router 通过 `KeyManager` 管理运行时状态（不持久化）
- 健康检查结果通过回调返回给 UI（不持久化）

### 7.5 迁移钩子位置

**怎么做**：在项目打开流程中插入迁移逻辑。

**为什么**：确保在用户看到任何 UI 之前，配置已经是最新的 V2 格式。

```ts
// 项目打开流程中的迁移钩子位置

async function openProject(projectPath: string): Promise<void> {
  // 1. 加载项目
  const configPath = path.join(projectPath, '.studio', 'config.json');

  // 2. 【迁移钩子】在此处插入
  await migrateWithBackup(configPath);

  // 3. 加载迁移后的配置
  const config = loadConfig(configPath);

  // 4. 【健康检查钩子】在此处插入
  const healthReport = await configStore.checkConfigHealth();

  // 5. 根据检查结果决定是否展示诊断面板
  if (healthReport.overallStatus === 'block') {
    showConfigDiagnosticPanel(healthReport);
  } else if (healthReport.overallStatus === 'warning') {
    showHealthWarningToast(healthReport);
  }

  // 6. 正常进入项目
  initProject(config);
}
```

### 7.6 实施优先级建议
