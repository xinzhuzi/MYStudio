## 1. 错误处理总则

### 1.1 五条恢复原则

以下原则来自融合总计划 12.9 节，是所有错误处理设计的基础：

| # | 原则 | 说明 |
|---|------|------|
| 1 | **AI 生成失败不影响已成功素材** | 单项生成失败时，已成功下载/保存的素材保持不变；失败项可单独重试，无需重新生成全部。 |
| 2 | **TTS 失败不阻塞视频候选** | TTS 生成失败时，视频候选（无声）仍可正常合成和预览；但带对白的单镜合成（需要音画同步）被阻塞，直到 TTS 成功或用户手动上传音频替代。 |
| 3 | **单镜合成失败不影响其他镜头** | 每个镜头的合成是独立任务，某个镜头 FFmpeg 失败不会级联到其他镜头。已合成成功的镜头保持可用。 |
| 4 | **Timeline 导出失败保留诊断信息** | 导出失败时，系统保留：FFmpeg 命令摘要、输入资产列表（含路径和文件大小）、完整 stderr 日志。便于开发者定位问题。 |
| 5 | **取消导出后状态回到 canceled** | 用户主动取消导出后，所有相关任务状态统一设为 `canceled`，临时文件按清理策略处理，UI 回到可重新导出状态。 |

### 1.2 错误分级

所有错误码必须归属于以下三个级别之一：

| 分级 | 标识 | 含义 | 典型场景 |
|------|------|------|----------|
| **Block** | `B` | 阻断整个流程，无法继续，需要人工干预 | FFmpeg 未安装、项目文件损坏、磁盘空间不足 |
| **Recoverable** | `R` | 可恢复，允许重试；不影响已成功部分 | AI API 超时、单镜头合成失败、TTS 生成失败 |
| **Info** | `I` | 信息性提示，不阻断流程 | 旧项目自动迁移完成、非关键配置缺失使用默认值 |

分级决策树：

```
错误发生
├─ 能否自动恢复（重试/降级/跳过）？
│   ├─ 是 → Recoverable
│   └─ 否 → 是否阻断整个流程？
│       ├─ 是 → Block
│       └─ 否 → Info
```

## 2. 错误码目录

### 2.1 错误码格式

```
E_{MODULE}_{SHORT_NAME}(NNN)
```

- `MODULE`：模块缩写（STUDIO / RENDER / GEN / CONFIG / PROJECT）
- `NNN`：三位数字编号，按模块分段

### 2.2 IPC 错误码（001-099）

| 错误码 | 描述 | 分级 | Phase |
|--------|------|------|-------|
| `E_STUDIO_FFMPEG_NOT_FOUND(001)` | FFmpeg 可执行文件未找到或路径无效 | Block | 所有 |
| `E_STUDIO_FFMPEG_FAILED(002)` | FFmpeg 进程非零退出，stderr 已记录 | Recoverable | P3+ |
| `E_STUDIO_EXPORT_CANCELLED(003)` | 用户主动取消导出 | Info | P3+ |
| `E_STUDIO_PROBE_FAILED(004)` | 媒体探测失败（无法读取时长/尺寸/fps） | Recoverable | P3+ |
| `E_STUDIO_INVALID_PLAN(005)` | 导出计划校验失败（缺少必要字段或数据不一致） | Block | P3+ |

### 2.3 FFmpeg 渲染错误码（100-199）

| 错误码 | 描述 | 分级 | Phase |
|--------|------|------|-------|
| `E_RENDER_IMAGE_FAILED(101)` | 图片转视频失败（格式不支持/分辨率异常） | Recoverable | P3+ |
| `E_RENDER_VIDEO_FAILED(102)` | 视频处理失败（编解码器不支持/文件损坏） | Recoverable | P3+ |
| `E_RENDER_SUBTITLE_FAILED(103)` | 字幕渲染失败（SRT 格式错误/字体缺失） | Recoverable | P3+ |
| `E_RENDER_AUDIO_MIX_FAILED(104)` | PCM 混音失败（采样率不匹配/声道异常） | Recoverable | P3+ |
| `E_RENDER_CONCAT_FAILED(105)` | 片段拼接失败（编码参数不一致/文件缺失） | Recoverable | P3+ |
| `E_RENDER_EXPORT_FAILED(106)` | 最终导出失败（容器封装错误/IO 错误） | Block | P3+ |

## 3. 场景化恢复流程

每个场景包含四个要素：
- **触发条件**：什么情况下触发此场景
- **现象**：用户看到的表现
- **恢复步骤**：系统自动执行和用户手动操作的步骤
- **标记**：自动 / 手动 / 混合

### 场景 1：AI 图片/视频生成失败

| 项目 | 内容 |
|------|------|
| **触发条件** | AI API 返回错误、超时、或网络中断 |
| **现象** | 生成进度停留在某项，该项状态变为 `failed`；已成功素材保持 `completed` |
| **恢复步骤** | 1. 系统自动标记失败项为 `failed`，保留已成功素材<br>2. 用户点击"重试失败项"按钮<br>3. 如持续失败，用户可在设置中切换模型（如从 DALL-E 切换到 Stable Diffusion）<br>4. 重新生成仅针对 `failed` 状态的项 |
| **标记** | 混合（自动标记 + 手动重试/切换模型） |

### 场景 2：TTS 生成失败

| 项目 | 内容 |
|------|------|
| **触发条件** | TTS API 调用失败、音频文件下载异常、或音频格式不符合预期 |
| **现象** | 该角色对白的 TTS 状态变为 `failed`；无声视频候选仍可预览 |
| **恢复步骤** | 1. 系统自动标记 TTS 为 `failed`，不阻塞无声视频合成<br>2. 带对白的单镜合成被阻塞（需要音画同步）<br>3. 用户可选择：重试 TTS / 手动上传音频文件替代 / 跳过对白使用无声版本 |
| **标记** | 混合（自动标记 + 手动选择恢复方式） |

### 场景 3：单镜合成失败

| 项目 | 内容 |
|------|------|
| **触发条件** | FFmpeg 处理单个镜头时返回非零退出码 |
| **现象** | 该镜头状态变为 `failed`，其他镜头不受影响 |
| **恢复步骤** | 1. 系统自动记录 FFmpeg stderr 到日志<br>2. 系统保留该镜头的全部输入素材（图片/视频/TTS音频/字幕）<br>3. 用户查看错误详情后可调整参数（分辨率/编码/时长）<br>4. 点击"重试此镜头"重新合成 |
| **标记** | 混合（自动记录 + 手动调整参数重试） |

### 场景 4：整集拼接缺镜头

| 项目 | 内容 |
|------|------|
| **触发条件** | 执行整集 concat 时，部分镜头未合成完成（状态非 `ready`） |
| **现象** | 拼接进度报告缺失镜头列表；仅使用 `ready` 状态的候选进行拼接 |
| **恢复步骤** | 1. 系统自动跳过非 `ready` 镜头，仅拼接可用片段<br>2. 导出报告中列出缺失镜头的序号和标题<br>3. 用户可补全失败镜头后重新执行整集拼接 |
| **标记** | 自动（拼接）+ 手动（补全后重拼） |

### 场景 5：多轨导出失败（FFmpeg stderr 分析）

| 项目 | 内容 |
|------|------|
| **触发条件** | LTX 风格多轨 Timeline 导出时 FFmpeg 返回错误 |
| **现象** | 导出进度中断，日志中包含 FFmpeg 错误信息 |
| **恢复步骤** | 1. 系统自动解析 FFmpeg stderr，提取关键错误信息<br>2. 系统保留完整的 filter graph 描述和输入资产列表<br>3. 系统保留组装好的完整 FFmpeg 命令行（便于终端手动调试）<br>4. 用户可在"高级调试"面板查看命令并复制到终端执行<br>5. 用户可逐步调试（先单轨 → 再叠加） |
| **标记** | 自动（诊断信息收集）+ 手动（调试修复） |

### 场景 6：导出取消后的状态回滚

| 项目 | 内容 |
|------|------|
| **触发条件** | 用户点击"取消导出"按钮 |
| **现象** | 导出进度立即停止，UI 回到可操作状态 |
| **恢复步骤** | 1. 系统向 FFmpeg 进程发送 SIGTERM（macOS/Linux）或 TerminateProcess（Windows）<br>2. 等待最多 5 秒让进程正常退出，超时则强制终止<br>3. 清理当前导出任务产生的临时文件<br>4. 将导出状态设为 `canceled`<br>5. UI 恢复到导出前的状态，可重新发起导出 |
| **标记** | 自动 |

### 场景 7：项目文件损坏或版本不兼容

| 项目 | 内容 |
|------|------|
| **触发条件** | 打开项目时 JSON 解析失败，或版本号高于当前应用支持的版本 |
| **现象** | 打开项目失败，弹出错误对话框 |
| **恢复步骤** | 1. 系统检测到错误后，先提示用户备份原文件<br>2. 若为版本不兼容：尝试自动执行数据迁移（降级兼容）<br>3. 若为文件损坏：尝试从自动备份恢复（最近一次成功保存）<br>4. 迁移/恢复成功后提示用户检查数据完整性<br>5. 若自动恢复失败：提示用户手动修复或联系支持 |
| **标记** | 混合（自动迁移/恢复 + 手动确认） |

### 场景 8：FFmpeg 未安装或版本不兼容

| 项目 | 内容 |
|------|------|
| **触发条件** | 应用启动或首次导出时，FFmpeg 路径检测失败或版本低于最低要求 |
| **现象** | 导出相关功能按钮禁用或点击后弹出安装引导 |
| **恢复步骤** | 1. 系统自动检测 FFmpeg 路径（PATH / 自定义路径 / 内置路径）<br>2. 若未找到：弹出引导对话框，提供下载链接和安装说明<br>3. 若版本过低：提示最低版本要求，提供更新链接<br>4. 用户安装/更新后，点击"重新检测"验证 |
| **标记** | 自动（检测）+ 手动（安装） |

### 场景 9：模型 Key 额度用尽

| 项目 | 内容 |
|------|------|
| **触发条件** | AI API 返回 429（速率限制）或 402（余额不足） |
| **现象** | 所有 AI 生成任务暂停，队列中的任务保持 `pending` |
| **恢复步骤** | 1. 系统自动标记当前任务为 `failed`（错误码 `E_GEN_API_QUOTA_EXCEEDED`）<br>2. 系统暂停队列中剩余任务，不逐一发送请求<br>3. 用户可选择：<br>   - 切换到备用 API Key<br>   - 切换到其他供应商（如 OpenAI → Azure OpenAI）<br>   - 暂停队列等待额度恢复（部分 API 速率限制会在分钟级恢复）<br>4. 恢复后点击"继续生成"重新启动队列 |
| **标记** | 自动（暂停队列）+ 手动（切换 Key/供应商/恢复） |

### 场景 10：磁盘空间不足

| 项目 | 内容 |
|------|------|
| **触发条件** | 导出前磁盘空间检查不通过（预估所需空间 > 可用空间） |
| **现象** | 导出按钮点击后弹出空间不足提示 |
| **恢复步骤** | 1. 导出前系统自动估算所需磁盘空间（基于素材大小 × 1.5 安全系数）<br>2. 若空间不足：提示当前可用空间和所需空间<br>3. 提供清理建议：<br>   - 清理临时文件（`.my-studio-tmp/`）<br>   - 清理已完成项目的缓存<br>   - 更改导出目录到其他磁盘<br>4. 用户清理后重新尝试导出 |
| **标记** | 自动（检查）+ 手动（清理/更换目录） |

### 场景 11：ComfyUI 工作流执行超时

| 项目 | 内容 |
|------|------|
| **触发条件** | ComfyUI 工作流执行时间超过预设超时（默认5分钟） |
| **现象** | GenerationJob 状态变为 `failed`，错误码 E_WORKFLOW_TIMEOUT |
| **恢复步骤** | 1. 系统自动终止超时的工作流进程<br>2. 保留工作流输入参数和部分输出（如有）<br>3. 用户可选择：增加超时时间 / 简化工作流 / 切换到轻量模型 |
| **标记** | 自动（终止）+ 手动（调整参数） |

### 场景 12：HTML 模板渲染失败

| 项目 | 内容 |
|------|------|
| **触发条件** | Playwright/Electron 渲染 HTML 模板时崩溃或超时 |
| **现象** | 模板预览为空白或报错 E_TEMPLATE_RENDER_FAILED |
| **恢复步骤** | 1. 系统回退到无模板模式（直接输出原始素材）<br>2. 记录模板渲染错误日志（含模板路径和参数）<br>3. 用户可检查模板语法或切换到其他模板 |
| **标记** | 自动（回退）+ 手动（修复模板） |

## 4. 用户可见错误消息规范

### 4.1 消息格式

所有面向用户的错误消息遵循统一格式：

```
[错误码] 简短描述

建议操作：具体的操作指引
```

示例：

```
[E_GEN_TIMEOUT(204)] AI 生成请求超时

建议操作：请检查网络连接后点击"重试"。如持续超时，可在设置中切换模型。
```

### 4.2 建议操作类型

| 类型 | 标识 | 说明 | 示例 |
|------|------|------|------|
| 可重试 | `RETRY` | 用户可直接重试，无需额外操作 | 网络超时、临时服务不可用 |
| 需配置 | `CONFIG` | 需要在设置中修改配置 | API Key 无效、模型未绑定 |
| 需安装 | `INSTALL` | 需要安装或更新外部依赖 | FFmpeg 未找到、版本过低 |
| 需清理 | `CLEANUP` | 需要清理磁盘空间或临时文件 | 磁盘空间不足 |

### 4.3 中英双语预留

错误消息在代码中采用 i18n key 方式管理：

```typescript
// 错误消息定义示例
```

## 5. 临时文件清理策略

### 5.1 命名规则

所有临时文件统一存放在 `.my-studio-tmp/` 目录下，按任务和镜头组织：

```
.my-studio-tmp/
├── {taskId}/
│   ├── {segmentIndex}_input.mp4        # 输入视频
│   ├── {segmentIndex}_audio.pcm        # TTS 音频
│   ├── {segmentIndex}_subtitle.srt     # 字幕文件
│   ├── {segmentIndex}_composed.mp4     # 合成结果
│   └── {segmentIndex}_meta.json        # 元数据
├── export/
│   ├── {exportId}/
│   │   ├── concat_list.txt             # FFmpeg concat 文件列表
│   │   ├── filter_graph.txt            # filter_complex 描述
│   │   ├── ffmpeg_cmd.sh               # 完整命令行（可调试）
│   │   └── output.mp4                  # 导出产物
│   └── index.json                      # 导出临时文件索引
└── cleanup.lock                         # 清理锁文件
```

### 5.2 清理时机

| 时机 | 触发方式 | 规则 |
|------|----------|------|
| **正常完成** | 合成/导出完成回调 | 标记临时文件为可清理（设置 `cleanable: true`） |
| **应用启动** | 应用启动时自动扫描 | 清理超过 24 小时的未完成任务临时文件 |
| **定期清理** | 后台定时任务（每小时一次） | 清理标记为可清理超过 7 天的临时文件 |
| **用户手动** | 设置界面"清理缓存"按钮 | 清理所有可清理的临时文件 |

### 5.3 清理安全机制

```
清理前检查：
1. 获取 cleanup.lock（防止并发清理）
2. 检查文件是否被当前活跃任务引用
3. 检查文件创建时间是否超过阈值
4. 执行清理
5. 更新索引文件
6. 释放 cleanup.lock
```

安全原则：
- 正在使用的文件（被活跃任务引用）**永远不会被清理**
- 清理操作记录日志，便于排查
- 清理失败（权限问题等）不影响主流程

### 5.4 导出临时文件索引

索引文件 `export/index.json` 结构：

```json
{
  "version": 1,
  "exports": {
    "export_20260527_001": {
      "createdAt": "2026-05-27T10:30:00Z",
      "status": "completed",
      "cleanable": true,
      "cleanableSince": "2026-05-27T10:35:00Z",
      "files": [
        ".my-studio-tmp/export/export_20260527_001/concat_list.txt",
        ".my-studio-tmp/export/export_20260527_001/output.mp4"
      ],
      "totalSize": 52428800
    }
  }
}
```

## 6. 测试策略

### 6.1 测试分层总览

```
┌─────────────────────────────────────────┐
│            E2E Tests                     │  ← 端到端闭环验证
│  ┌─────────────────────────────────────┐ │
│  │        FFmpeg Smoke Tests           │ │  ← FFmpeg 真实执行验证
│  │  ┌─────────────────────────────────┐│ │
│  │  │      Integration Tests          ││ │  ← 跨模块交互验证
│  │  │  ┌─────────────────────────────┐││ │
│  │  │  │      Unit Tests             │││ │  ← 纯函数/逻辑验证
│  │  │  └─────────────────────────────┘││ │
│  │  └─────────────────────────────────┘│ │
│  └─────────────────────────────────────┘ │
└─────────────────────────────────────────┘
```

运行频率：
- **单元测试**：每次提交自动运行（CI）
- **集成测试**：每次 PR 合并前运行
- **FFmpeg Smoke**：每日构建 + 发布前
- **E2E**：版本发布前 + 重大功能变更后

### 6.2 单元测试覆盖要求

#### 6.2.1 纯函数 — 100% 覆盖

以下模块必须达到 100% 行覆盖率：

| 模块 | 测试要点 | 示例用例 |
|------|----------|----------|
| Timeline flatten | 多轨展开为单轨序列 | 2轨重叠 → 正确的 Z-order 和时间切片 |
| Filter graph 构建 | FFmpeg filter_complex 字符串生成 | 2视频+1音频+字幕 → 正确的 filter 语法 |
| 字幕提取优先级 | 多来源字幕的优先级排序 | 内嵌字幕 > SRT 文件 > 对白文本 |
| PCM 混音计算 | 采样率/声道对齐与音量混合 | 44.1kHz + 48kHz → 重采样后叠加 |

#### 6.2.2 Store Action — 关键路径覆盖

| 测试场景 | 覆盖内容 |
|----------|----------|
| 新旧字段兼容 | 读取缺少新字段的旧 Store 数据，验证默认值正确填充 |
| 默认值填充 | 新增配置项在旧数据中不存在时，使用合理的默认值 |
| 状态流转 | `pending → processing → completed/failed/canceled` 每条路径有测试 |

#### 6.2.3 类型守卫 — 迁移路径覆盖

每个项目版本迁移路径必须有对应测试：

```typescript
// 迁移测试示例
describe('Project Migration', () => {
  it('should migrate v1 project to v2 with default ttsEngine', () => {
    const v1Project = { version: 1, segments: [...] };
    const migrated = migrateProject(v1Project);
    expect(migrated.version).toBe(2);
    expect(migrated.ttsEngine).toBe('default');
    expect(migrated.segments).toEqual(v1Project.segments);
  });
});
```

### 6.3 集成测试场景

集成测试验证跨模块交互的正确性，使用 Mock 替代外部依赖（AI API / FFmpeg）。

#### 场景 A：IPC 端到端

```
Renderer → Preload → Main → FFmpeg(Mock) → 结果回到 Renderer
```

| 测试用例 | 验证点 |
|----------|--------|
| 渲染进程发起合成请求 | IPC 通道正确传递参数 |
| Main 进程调用 FFmpeg | 参数组装正确、进度回调正常 |
| FFmpeg 返回成功 | 结果文件路径正确回传到 Renderer |
| FFmpeg 返回失败 | 错误码和 stderr 正确传递到 Renderer |

#### 场景 B：配置 → 任务绑定 → 执行 → 状态更新

| 测试用例 | 验证点 |
|----------|--------|
| 配置绑定后生成图片 | 使用正确的模型和参数 |
| 切换模型后重新生成 | 新模型参数生效 |
| 绑定缺失时发起任务 | 返回 `E_CONFIG_NO_BINDING` 错误 |

#### 场景 C：项目打开 → 旧数据迁移 → 健康检查

| 测试用例 | 验证点 |
|----------|--------|
| 打开 V1 项目 | 自动迁移完成，所有字段可读 |
| 打开当前版本项目 | 无迁移执行，直接加载 |
| 打开损坏项目 | 返回 `E_PROJECT_FILE_CORRUPTED` |
| 迁移后资产检查 | 引用的资产文件存在性检查 |

### 6.4 FFmpeg Smoke Test 矩阵

Smoke test 使用真实 FFmpeg 执行，验证核心渲染管线。测试素材存放在 `test/fixtures/media/` 目录。

| # | 测试项 | 输入 | 预期输出 | 超时 |
|---|--------|------|----------|------|
| 1 | 图片转视频（竖屏） | 1080x1920 PNG, 3秒 | 1080x1920 MP4, H.264 | 30s |
| 2 | 图片转视频（横屏） | 1920x1080 PNG, 3秒 | 1920x1080 MP4, H.264 | 30s |
| 3 | 视频 + TTS + SRT 合成 | MP4 + PCM + SRT | 合成 MP4, 音画同步 | 60s |
| 4 | 5 片段 concat | 5 个 MP4 | 拼接后完整视频 | 60s |
| 5 | 10 片段 concat | 10 个 MP4 | 拼接后完整视频 | 120s |
| 6 | LTX 多轨导出（2轨重叠） | 2 视频 + overlay filter | 叠加后 MP4 | 60s |
| 7 | 导出取消 | 运行 3 秒后取消 | 进程终止、临时文件清理 | 15s |
| 8 | FFmpeg 路径探测 | 系统 PATH + 自定义路径 | 版本号 ≥ 最低要求 | 5s |

| 10 | PCM 混音（2轨叠加） | 2 个 PCM 文件 | 混音后 PCM, 音量叠加正确 | 15s |

Smoke test 判定标准：
- 所有测试项必须在真实 FFmpeg 环境下通过
- 不允许 Mock FFmpeg 行为
- 失败时输出完整的 FFmpeg 命令和 stderr

### 6.5 E2E 测试计划

E2E 测试验证用户完整工作流，在发布前和重大功能变更后执行。


验证点：
1. V1 项目文件打开无错误
2. 所有旧字段正确显示
3. 新增字段使用合理默认值
