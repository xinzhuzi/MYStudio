### 1.4 已知构建约束

- `main.ts` 单文件超过 2000 行，承担全部 FFmpeg 渲染编排逻辑
- 无 electron-builder 配置文件，尚未建立正式打包流程
- 缺少 CI/CD 流水线

### 1.5 最低系统需求

| 维度 | 最低配置 | 推荐配置 | 说明 |
|------|---------|---------|------|
| 操作系统 | macOS 12+ / Windows 10+ / Ubuntu 20.04+ | macOS 14+ / Windows 11 | Electron 30 的最低要求 |
| CPU | 4核 x86_64 或 Apple Silicon | 8核+ | FFmpeg 编码和音频混音消耗CPU |
| 内存 | 8 GB | 16 GB+ | PCM 混音和视频缓存需要内存 |
| 磁盘 | 2 GB 可用空间 | 10 GB+ | 素材/缓存/导出产物占用空间大 |
| GPU | 无硬性要求（AI通过API） | NVIDIA GPU（本地ComfyUI） | 仅本地AI生成需要GPU |
| FFmpeg | 5.0+ (LGPL) | 6.0+ (GPL，含libx264) | 系统安装或应用内置 |
| Node.js | 18.x（开发构建） | 20.x LTS | 仅开发环境需要 |
| 网络 | 按需（使用云端API时） | 稳定宽带 | 本地模式可离线使用 |

### 1.6 首次运行体验

应用首次启动应执行以下检查并引导用户：

1. **FFmpeg 检测**：扫描系统 PATH 和内置路径，如未找到则弹窗引导安装
2. **配置初始化**：创建默认 `VendorConfig`，引导用户配置至少一个 AI 供应商
3. **项目目录**：创建 `app-resources/`（内置模板）和 `user-data/`（用户目录）
4. **示例项目**：可选，提供"3镜头短剧 Demo"引导用户走通完整流程

## 2. 代码模块拆分路线

### 2.1 当前问题

`src/electron/main.ts` 承担所有 FFmpeg 渲染逻辑，导致：
- 难以单独测试各子模块
- IPC 注册与业务逻辑耦合
- 代码审查困难，变更风险高

### 2.2 目标结构

```
src/electron/
  main.ts                        -- 入口，只注册IPC，不包含业务逻辑
  export/
    ffmpeg-utils.ts              -- FFmpeg路径探测、进程管理、取消、媒体探测
    timeline.ts                  -- 多轨clip到segment的flatten
    video-filter.ts              -- FFmpeg filter graph构建
    audio-mix.ts                 -- PCM音频混音
    export-handler.ts            -- IPC编排、临时文件、输出、错误处理
  studio-render/
    template-renderer.ts         -- HTML模板预览渲染
    resource-scanner.ts          -- 资源发现扫描

src/shared/types/
  studio.ts                      -- 工作流类型（Workflow, Track, Clip, Effect）
  export.ts                      -- 导出计划类型（ExportPlan, Segment, FilterGraph）
  template.ts                    -- 模板相关类型（Template, TemplateSlot）
```

### 2.3 拆分步骤（保持 IPC 兼容）

> 原则：每一步完成后项目可正常 `npm run dev`，不中断开发。

#### 步骤 1：抽取 ffmpeg-utils

```
main.ts 中的函数移入 export/ffmpeg-utils.ts：
  - findFFmpegPath()        -- 路径探测
  - spawnFFmpegProcess()    -- 进程创建
  - cancelFFmpegProcess()   -- 进程取消
  - probeMedia()            -- 媒体信息探测
```

- main.ts 改为 `import { ... } from './export/ffmpeg-utils'`
- 验证：`npm run dev` + 导出一个视频确认功能正常

#### 步骤 2：抽取 export-handler

```
main.ts 中的函数移入 export/export-handler.ts：
  - handleExport()          -- IPC handler 主入口
  - createTempDir()         -- 临时文件管理
  - finalizeOutput()        -- 输出文件处理
  - handleExportError()     -- 错误处理与清理
```

- main.ts 中只保留 `ipcMain.handle('export', exportHandler.handleExport)`
- 验证：完整导出流程测试

#### 步骤 3：抽取 timeline / video-filter / audio-mix

```
export-handler.ts 中调用的函数拆入独立文件：
  - timeline.ts     -- flattenTracks(), buildSegments()
  - video-filter.ts -- buildFilterGraph(), composeFilters()
  - audio-mix.ts    -- mixPCMStreams(), normalizeAudio()
```

- export-handler.ts 改为 import 这三个模块
- 验证：全量回归测试

### 2.4 类型文件拆分

从 `src/shared/types.ts` 拆分为：

| 文件 | 包含类型 | 说明 |
|------|---------|------|
| `studio.ts` | `Workflow`, `Track`, `Clip`, `Effect`, `ProjectMeta` | 工作流核心模型 |
| `export.ts` | `ExportPlan`, `Segment`, `FilterGraph`, `ExportProgress` | 导出管线类型 |
| `template.ts` | `Template`, `TemplateSlot`, `RenderConfig` | 模板引擎类型 |

### 2.5 Store 拆分参考

借鉴 moyin 多 store 模式，建议将单一 store 拆为：

| Store | 职责 | 关键 state |
|-------|------|-----------|
| `useProjectStore` | 项目元数据、时间线 | `project`, `tracks`, `clips` |
| `useExportStore` | 导出状态、进度 | `exportPlan`, `progress`, `isExporting` |
| `useResourceStore` | 资源库管理 | `resources`, `templates`, `effects` |

## 3. Electron Builder 配置

### 3.1 基础配置结构

创建 `electron-builder.yml`：

```yaml
appId: com.manying.studio
productName: Manying Studio
copyright: Copyright © 2026 Manying Studio

directories:
  output: dist
  buildResources: build

files:
  - out/**/*
  - "!out/**/*.map"

extraResources:
  - from: app-resources
    to: app-resources
    filter:
      - "**/*"
  - from: build/ffmpeg
    to: ffmpeg
    filter:
      - "**/*"

asar: true
asarUnpack:
  - "**/*.node"           # 原生模块
  - "**/ffmpeg/**"        # FFmpeg二进制不打包进ASAR
```

### 3.2 macOS 差异配置

```yaml
# electron-builder.yml (mac部分)
mac:
  target:
    - target: dmg
      arch:
        - universal
  category: public.app-category.productivity
  icon: build/icon.icns
  hardenedRuntime: true
  gatekeeperAssess: false
  entitlements: build/entitlements.mac.plist
  entitlementsInherit: build/entitlements.mac.plist
  # 代码签名（需要Apple Developer证书）
  identity: "Developer ID Application: Your Name (TEAMID)"

dmg:
  sign: true
  contents:
    - x: 130
      y: 220
    - x: 410
      y: 220
      type: link
      path: /Applications

# 公证（notarize）- macOS 10.15+ 必需
afterSign: build/notarize.js
```

公证脚本 `build/notarize.js` 关键逻辑：

```javascript
const { notarize } = require('@electron/notarize');
exports.default = async function notarizing(context) {
  const { electronPlatformName, appOutDir } = context;
  if (electronPlatformName !== 'darwin') return;
  const appName = context.packager.appInfo.productFilename;
  const appPath = `${appOutDir}/${appName}.app`;
  await notarize({
    appPath,
    appleId: process.env.APPLE_ID,
    appleIdPassword: process.env.APPLE_APP_PASSWORD,
    teamId: process.env.APPLE_TEAM_ID,
  });
};
```

| 内容 | ASAR 内 | ASAR 外 | 原因 |
|------|---------|---------|------|
| 应用 JS 代码 | Yes | — | 标准做法 |

| 应用图标 | — | Yes | electron-builder 从外部引用 |

## 4. FFmpeg 内置策略

### 4.1 方案对比

| 维度 | 系统回退 | 内置分发 |
|------|---------|---------|
| 安装包体积 | 小（~80MB） | 大（macOS +60MB, Windows +80MB） |
| 用户门槛 | 需额外安装 FFmpeg | 开箱即用 |
| 版本控制 | 不可控，依赖系统包 | 精确控制 |
| 许可证风险 | 低 | 需仔细处理 |
| 维护成本 | 低 | 需随平台更新二进制 |
| 推荐方案 | 备选/开发者模式 | **主方案** |

**结论：采用"内置为主、系统回退为辅"的混合策略。**

### 4.2 许可证考量

| 许可证 | 条件 | 适用场景 | 本项目选择 |
|--------|------|---------|-----------|
| LGPL 2.1+ | 动态链接即可闭源 | 默认配置 | **首选** |
| GPL 2.0+ | 必须开源 | 启用 libx264 等GPL库 | 避免使用 |
| 静态链接 | 必须提供目标文件或开源 | 单一可执行文件 | 避免 |
| 动态链接 | LGPL 兼容闭源 | .dylib / .dll | **采用** |

**核心原则**：
- 使用 FFmpeg LGPL 构建版本（不启用 --enable-gpl）
- 视频编码优先使用 libvpx-vp9 / libaom-av1（BSD/Apache 许可）
- 如需 H.264，使用 OpenH264（思科开源，BSD许可）而非 libx264（GPL）
- 以动态链接方式分发（.dylib / .dll / .so）

### 4.3 各平台 FFmpeg 分发方式

#### macOS

```
app-resources/
  ffmpeg/
    mac/
      ffmpeg          # Universal binary (arm64 + x64)
      ffprobe         # Universal binary
```

- 构建：从 ffmpeg.org 下载 LGPL 静态构建，或使用 Homebrew 的 LGPL 配方自行编译
- 签名：需要对 ffmpeg/ffprobe 进行 ad-hoc 签名或使用开发者证书签名
- 备选：检测到系统无内置 FFmpeg 时，提示用户 `brew install ffmpeg`

#### Windows

```
app-resources/
  ffmpeg/
    win/
      ffmpeg.exe
      ffprobe.exe
```

- 来源：使用 Gyan.dev 提供的 LGPL shared 或 essentials 构建
- 路径：`path.join(process.resourcesPath, 'ffmpeg', 'win', 'ffmpeg.exe')`
- 签名：使用代码签名证书签名 .exe

#### Linux

```
app-resources/
  ffmpeg/
    linux/
      ffmpeg
      ffprobe
```

- 来源：使用 Linux 静态构建（John Van Sickle 提供）
- 备选：检测到无内置时提示 `sudo apt install ffmpeg` / `sudo dnf install ffmpeg`
- AppImage 场景下内置二进制需要设置 execute 权限

### 4.4 FFmpeg 检测与引导安装流程

```
应用启动
  │
  ├─ 检测内置 FFmpeg（app-resources/ffmpeg/{platform}/）
  │   ├─ 存在 → 验证版本 ≥ 最低要求 → 使用
  │   └─ 不存在 → 进入引导流程
  │
  ├─ 检测系统 FFmpeg（which/where ffmpeg）
  │   ├─ 存在 → 验证版本 → 使用（显示提示：建议使用内置版本）
  │   └─ 不存在 → 进入引导流程
  │
  └─ 引导流程：
      ├─ macOS:  弹窗提示 "brew install ffmpeg" 或下载内置包
      ├─ Windows: 自动下载内置包到 userData/ffmpeg/
      └─ Linux:  弹窗提示 "sudo apt install ffmpeg" 或下载内置包
```

### 4.5 FFmpeg 版本要求

| 项目 | 最低版本 | 推荐版本 |
|------|---------|---------|
| FFmpeg | 5.0 | 6.1+ |
| 需要的编码器 | libvpx-vp9, libaom-av1, pcm_s16le | 同左 |
| 需要的滤镜 | overlay, amix, adelay, pad | 同左 |
| 探测工具 | ffprobe | 同 FFmpeg 版本 |

## 5. 模板与资源打包

### 5.1 资源目录规划

```
应用资源（只读，随应用分发）:
  app-resources/
    templates/              # 内置模板
      basic/
        manifest.json
        preview.png
        template.html
      cinematic/
        ...
    workflows/              # 内置工作流预设
      default.json
      quick-export.json
    effects/                # 内置特效
      transitions/
      filters/

用户数据（可读写）:
  userData/                 # electron.app.getPath('userData')
    projects/               # 用户项目文件
    custom-templates/       # 用户自定义模板
    custom-effects/         # 用户自定义特效
    cache/                  # 缓存（可清理）
    ffmpeg/                 # 首次下载的 FFmpeg（如未内置）
    downloads/              # 按需下载的资源包
```

### 5.2 ASAR 内 vs 外部资源

| 资源类型 | 位置 | 访问方式 |
|---------|------|---------|
| 内置模板 | `process.resourcesPath + '/app-resources/templates'` | 只读 |
| 内置特效 | `process.resourcesPath + '/app-resources/effects'` | 只读 |
| 用户模板 | `userData + '/custom-templates'` | 读写 |
| 项目文件 | `userData + '/projects'` | 读写 |
| 缓存 | `userData + '/cache'` | 读写，可清理 |
| 大型资源包 | `userData + '/downloads'` | 按需下载 |

### 5.3 大型特效资源包按需下载策略

```
用户选择特效
  │
  ├─ 检查本地是否已下载（userData/downloads/{pack-id}/）
  │   ├─ 已下载 → 直接使用
  │   └─ 未下载 → 显示下载对话框
  │       ├─ 确认下载 → 从CDN下载zip → 解压到userData → 验证完整性
  │       └─ 取消 → 使用默认特效
  │
  └─ 下载完成后注册到 resourceStore
```

## 6. 自动更新

### 6.1 electron-updater 配置

```yaml
# electron-builder.yml 中追加
publish:
  provider: github
  owner: manying-studio
  repo: manying-studio
  releaseType: release
```

主进程更新检查代码：

```typescript
import { autoUpdater } from 'electron-updater';
import log from 'electron-log';

autoUpdater.logger = log;
autoUpdater.autoDownload = false;       // 不自动下载，先提示用户
autoUpdater.autoInstallOnAppQuit = true; // 退出时自动安装

// 检查更新
autoUpdater.checkForUpdates();

// 发现新版本 → 通知渲染进程
autoUpdater.on('update-available', (info) => {
  mainWindow.webContents.send('update-available', info);
});

// 下载完成 → 提示安装
autoUpdater.on('update-downloaded', (info) => {
  mainWindow.webContents.send('update-downloaded', info);
});

// IPC: 用户确认下载
ipcMain.handle('download-update', () => autoUpdater.downloadUpdate());

// IPC: 用户确认安装
ipcMain.handle('install-update', () => autoUpdater.quitAndInstall());
```

### 6.2 更新通道

| 通道 | 用途 | 触发条件 | 稳定性 |
|------|------|---------|--------|
| stable | 正式发布 | 手动触发 | 已全面测试 |
| beta | 公开测试 | 每周构建 | 功能完整，可能有已知问题 |
| canary | 开发预览 | 每次合并main | 最新功能，不保证稳定 |

通道切换通过 `autoUpdater.channel` 控制：

```typescript
// 用户在设置中选择通道
const channel = userSettings.updateChannel; // 'stable' | 'beta' | 'canary'
autoUpdater.channel = channel;
autoUpdater.checkForUpdates();
```

### 6.3 更新检查时机

| 时机 | 行为 | 可配置 |
|------|------|--------|
| 应用启动 | 静默检查，有更新则通知 | 是（默认开启） |
| 每4小时 | 静默检查 | 是（默认关闭） |
| 手动检查 | 用户点击"检查更新" | 始终可用 |
| 安装后首次启动 | 跳过检查（避免刚更新又提示） | 否 |

### 6.4 FFmpeg 版本更新策略

```
场景：应用更新后内置 FFmpeg 版本也更新了

应用启动时：
  1. 读取内置 FFmpeg 版本（app-resources/ffmpeg/{platform}/VERSION）
  2. 读取上次使用的 FFmpeg 版本（userData/ffmpeg-version.json）
  3. 如果版本不同：
     - 更新 ffmpeg-version.json
     - 清除 FFmpeg 缓存（userData/cache/ffmpeg/）
     - 使用新版本
  4. 如果用户自定义安装了系统 FFmpeg：
     - 不覆盖，仅提示"当前使用系统 FFmpeg x.x，内置版本为 y.y"
```

## 7. 项目存储格式演进

### 7.1 版本号管理

项目文件 `project.json` 包含 `version` 字段：

```json
{
  "version": "1.0",
  "name": "我的漫剧",
  "tracks": [...],
  "meta": {...}
}
```

版本号规则：
- 主版本（X.0）：不兼容的格式变更
- 次版本（1.X）：兼容的扩展（新增字段，旧版本忽略即可）
- 初始版本：`"1.0"`

### 7.2 格式迁移钩子

打开项目时的迁移流程：

```
加载 project.json
  │
  ├─ 读取 version 字段
  │   ├─ 缺失 → 视为 "1.0"（初始格式）
  │   └─ 存在 → 解析版本号
  │
  ├─ 比较当前支持的最新版本（如 "1.3"）
  │   ├─ 相同 → 直接加载
  │   └─ 不同 → 执行迁移链
  │
  └─ 迁移链：
      project.json → 备份为 project.json.bak
      1.0 → 1.1 → 1.2 → 1.3（依次执行）
      迁移后写入 project.json
```

### 7.3 迁移函数注册机制

```typescript
// src/shared/migrations.ts

interface Migration {
  version: string;           // 从哪个版本
  targetVersion: string;     // 迁移到哪个版本
  migrate(data: any): any;   // 迁移函数
}

const migrations: Migration[] = [
  {
    version: '1.0',
    targetVersion: '1.1',
    migrate(data) {
      // V1.1: 新增 tracks[].clips[].effects 数组
      for (const track of data.tracks || []) {
        for (const clip of track.clips || []) {
          if (!clip.effects) clip.effects = [];
        }
      }
      data.version = '1.1';
      return data;
    }
  },
  {
    version: '1.1',
    targetVersion: '1.2',
    migrate(data) {
      // V1.2: meta 字段结构化
      if (!data.meta) data.meta = {};
      if (!data.meta.createdAt) data.meta.createdAt = new Date().toISOString();
      data.version = '1.2';
      return data;
    }
  },
  // ... 后续迁移
];

export function migrateProject(data: any): any {
  let current = data.version || '1.0';
  while (current !== LATEST_VERSION) {
    const migration = migrations.find(m => m.version === current);
    if (!migration) break;
    data = migration.migrate(data);
    current = data.version;
  }
  return data;
}
```

### 7.4 回滚保护

```typescript
import { readFileSync, writeFileSync, copyFileSync } from 'fs';

function loadProjectWithMigration(filePath: string) {
  // 1. 迁移前备份
  const bakPath = filePath + '.bak';
  copyFileSync(filePath, bakPath);

  // 2. 读取并迁移
  let data = JSON.parse(readFileSync(filePath, 'utf-8'));
  const needsMigration = (data.version || '1.0') !== LATEST_VERSION;

  if (needsMigration) {
    try {
      data = migrateProject(data);
      writeFileSync(filePath, JSON.stringify(data, null, 2));
    } catch (err) {
      // 3. 迁移失败 → 从备份恢复
      copyFileSync(bakPath, filePath);
      throw new Error(`项目迁移失败: ${err.message}`);
    }
  }

  return data;
}
```

| 1.1 | 新增 clip.effects 数组 | 向后兼容 |

## 9. 常见构建问题排查

| 问题 | 可能原因 | 排查步骤 |
|------|---------|---------|
| `npm run dev` 启动白屏 | Electron主进程启动失败 | 检查终端错误日志；确认electron-vite配置正确 |
| FFmpeg not found | 未安装或不在PATH中 | 运行 `ffmpeg -version`；检查 `findFFmpegPath()` 返回值 |
| 原生模块编译失败 | Node版本与Electron不匹配 | 运行 `npx electron-rebuild`；确认Node版本 ≥ 18 |
| 打包后ASAR内文件读不到 | 路径使用了 `__dirname` | 使用 `process.resourcesPath` 或 `app.getPath` |
| macOS 签名失败 | 缺少开发者证书 | 确认 `APPLE_ID` 等环境变量已设置；检查证书有效期 |
| Windows 安装器报毒 | 未签名或SmartScreen | 配置代码签名证书；提交Microsoft Defender审核 |
| `isolated-vm` 编译报错 | C++编译工具链缺失 | macOS: `xcode-select --install`；Windows: 安装VS Build Tools |
| 视频导出黑屏 | FFmpeg filter graph 错误 | 检查临时目录中的 filter_complex_script 文件；手动执行FFmpeg命令调试 |
| 项目文件损坏 | JSON解析失败 | 检查 `project.json.bak` 是否存在；手动修复JSON语法 |
| 内存占用过高 | PCM混音或大素材 | 检查素材分辨率是否超过建议值；减少并发导出数 |
